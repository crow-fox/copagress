# Copagress — Markdown タスク仕様

このドキュメントは Copagress が読み取る Markdown 形式の最終仕様を示します。

## 基本ルール（概要）

- ファイルごとに最も大きな単位で分ける（ファイル＝ワークスペース内の単位）。
- レベル1見出し (`#`) = プロジェクト名（project）
- レベル2見出し (`##`) = ステータス（status）
- レベル2 以下の段落はすべてタスクとして扱う（見出しレベル3 以降もタスク扱い）。
- 箇条書き（`-` `*` `+`）もタスクとして扱う。
- インデントは無視する：行頭の空白は削除（先頭の空白を詰める）。
- H1（`#`）が無い場合はファイル名（拡張子を除く）を `project` として扱う。
- H2（`##`）が無い場合は当該セクションの `status` を `未着手` として扱う。

## パースルール

1. ファイル全体を読み、`H1`（`# `）で区切られた各セクションを `project` として扱う。
2. 各 `project` 内で `H2`（`## `）を検出し、見出しテキストを `status` として扱う。
3. 各 `status` 配下のすべてのブロック（段落、見出しレベル3+, 箇条行）は個別のタスクとみなす。
4. 行頭の空白はトリム（削除）して `title` を得る。行内の空白は保持する。
5. ネスト（サブタスク）や追記事項はサポートしない。すべてフラットなタスク列とする。

### 補足: ファイル先頭に段落等がある場合の扱い

- ファイルの最初に `H1` より前に段落や箇条行が現れる場合の処理:
  - ファイル内に少なくとも1つ `H1` が存在する場合: その先頭の `H1`（最初の project）に属するタスクとして扱う。
  - ファイル内に `H1` が全く存在しない場合: ファイル名（拡張子を除く）を `project` として扱い、その下のタスクとして扱う。
  - いずれの場合も、該当するタスクの `status` は当該 `project` に `H2` が存在しない限り `未着手` を割り当てる。

補足（H1/H2 が省略されたファイルの扱い）:

- ファイルに `H1` が存在しない場合、ファイル名を `project` とみなして解析を行う（例: `daily.md` → project: "daily"）。
- ある `project` 内に `H2` が無い場合、該当 `project` のすべてのタスクは `status: "未着手"` として扱われる。

## Task 型（仕様）

```ts
type Task = {
  project: string; // H1 のテキスト
  status: string; // H2 のテキスト
  title: string; // タスクのテキスト（先頭空白は削除）
  row: number; // 元ドキュメント内の行番号（重複しないのでリスト表示のkeyにしようできる）　ソートにも利用可能（orderが代わり）
};
```

## 例

```md
# 日常a

## 未着手

- タスクA
- 段落としてのタスクB
- タスクC（見出しレベル3 もタスク）

## 実行中

- タスクD

## 保留

- タスクE（保留中の確認作業）

## 完了

- タスクF（完了した作業）

# 日常b

## 未着手

## 実行中

## 保留

## 完了
```

このファイルから抽出されるタスクの一部（概念）:

```ts
[
  // 日常a / 未着手
  { project: "日常a", status: "未着手", title: "タスクA", row: 4 },
  { project: "日常a", status: "未着手", title: "段落としてのタスクB", row: 5 },
  {
    project: "日常a",
    status: "未着手",
    title: "タスクC（見出しレベル3 もタスク）",
    row: 6,
  },

  // 日常a / 実行中
  { project: "日常a", status: "実行中", title: "タスクD", row: 9 },

  // 日常a / 保留
  {
    project: "日常a",
    status: "保留",
    title: "タスクE（保留中の確認作業）",
    row: 12,
  },

  // 日常a / 完了
  {
    project: "日常a",
    status: "完了",
    title: "タスクF（完了した作業）",
    row: 15,
  },
];
```

## 運用・補足

- ファイルごとに管理することで、プロジェクトの粒度を自由に分けられる（例: 日常.md, work.md）。
- 単一ファイル内で複数プロジェクトを管理する場合は `H1/H2` 構造に従うことでネスト不要で扱える。
- 保存ファイルの命名はシンプルに `<slug>.md`（例: `daily.md`, `仕事.md`）で良い。
- ステータス名の正規化やエイリアスは行わない（任意のテキストをそのまま `status` として扱う）。
- インポート／エクスポートや UI 操作の詳細は当面仕様から除外する（実装段階で検討）。
- `order` はあくまでパーサのスナップショット上での出現順を示すもので、永続的な識別子ではない。
